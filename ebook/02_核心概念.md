# 第 2 章：核心概念 — Session / Context / Tools / Permission 详解

## 理解 Claude Code 的四根支柱

Claude Code 的强大不是来自单一功能，而是四个核心概念的有机组合。理解了它们，你就掌握了 Claude Code 的底层逻辑，后续的技巧和高级用法都会水到渠成。

## 一、Session（会话）

### 什么是 Session？

Session 是你和 Claude Code 之间的一次完整对话。从你启动 `claude` 命令开始，到你退出或关闭终端结束。在一个 Session 中，Claude Code 会记住所有的对话历史、它读过的文件、执行过的命令。

```bash
# 启动一个新 Session
$ claude

# 在 Session 中的每次交互都共享上下文
> 读取 src/index.ts
> 把里面的 Express 改成 Fastify   # Claude 记得它刚读过什么
> 运行测试看看有没有问题         # Claude 知道它改了什么
```

### Session 的生命周期

1. **创建**：运行 `claude` 时自动创建
2. **活跃**：你和 Claude Code 交互期间
3. **暂停**：长时间无操作时 Session 会暂停，但状态保留
4. **恢复**：使用 `claude --resume` 恢复上一个 Session
5. **结束**：退出终端或超过 Session 时限

### 关键实践

- **一个任务一个 Session**：不要在同一个 Session 里混杂不相关的任务，这会污染上下文
- **善用 resume**：如果中途需要离开，直接关闭终端即可，下次 `claude --resume` 继续
- **Session ID**：每个 Session 都有唯一 ID，可以在 `~/.claude/projects/` 下找到历史记录

## 二、Context（上下文）

### Context 是什么？

Context 是 Claude Code 在当前 Session 中"知道"的所有信息。这包括：

- **对话历史**：你说了什么，它回复了什么
- **文件内容**：它读过的所有文件
- **命令输出**：它执行过的命令的结果
- **项目元数据**：CLAUDE.md、package.json、目录结构等

### Context Window 的限制

Claude 模型有上下文窗口限制（目前约 200K tokens）。虽然很大，但在处理大型项目时仍然需要注意：

```
# 上下文使用的粗略估算
1 个中等大小的源文件 ≈ 500-2000 tokens
1 次命令输出 ≈ 200-5000 tokens
对话历史会持续累积
```

### Context 管理策略

**CLAUDE.md 是你的秘密武器。** 在项目根目录放一个 `CLAUDE.md` 文件，Claude Code 每次启动时都会自动读取。这是你向 AI "介绍"项目的最佳方式：

```markdown
# CLAUDE.md

## 项目概述
这是一个基于 FastAPI 的电商后端，使用 PostgreSQL 数据库。

## 技术栈
- Python 3.12 + FastAPI
- SQLAlchemy 2.0 + Alembic
- Redis 缓存
- Docker Compose 本地开发

## 代码规范
- 使用 ruff 做代码格式化
- 类型注解必须完整
- 测试覆盖率要求 > 80%

## 常用命令
- `make dev` — 启动开发环境
- `make test` — 运行测试
- `make lint` — 代码检查
```

**分层 Context**：Claude Code 支持多级 CLAUDE.md：
- `~/.claude/CLAUDE.md` — 全局级别（你的个人偏好）
- `/project/CLAUDE.md` — 项目级别（项目规范）
- `/project/src/CLAUDE.md` — 目录级别（模块特定信息）

### Context 压缩

当上下文接近窗口限制时，Claude Code 会自动进行 context compaction——压缩早期的对话，保留关键信息。你也可以主动管理：

```
> /compact   # 手动触发上下文压缩
```

## 三、Tools（工具）

### Claude Code 的工具箱

Claude Code 不只是"聊天"——它能通过一系列内置工具直接操作你的开发环境：

| 工具 | 能力 | 示例 |
|------|------|------|
| **read** | 读取文件内容 | 读代码、配置文件、日志 |
| **write** | 创建或覆盖文件 | 写代码、生成配置 |
| **edit** | 精确编辑文件 | 替换特定代码段 |
| **exec** | 执行 shell 命令 | 运行测试、安装依赖、git 操作 |
| **web_search** | 搜索互联网 | 查文档、找解决方案 |
| **web_fetch** | 抓取网页内容 | 读取在线文档 |
| **browser** | 控制浏览器 | UI 测试、网页自动化 |

### 工具调用的透明性

Claude Code 的每一次工具调用都是透明的。你会在终端中看到它正在做什么：

```
🔧 Reading file: src/models/user.py
🔧 Executing: npm test
🔧 Writing file: src/models/user.py
```

这意味着你可以随时了解 AI 的行为，必要时中断或纠正。

### 扩展工具能力

通过 MCP（Model Context Protocol）和 Skills 系统，你可以为 Claude Code 添加自定义工具。这是 Claude Code 和其他 AI 工具的核心差异之一——它是**可扩展的**。第 4 章会详细讲解 Skills 系统。

## 四、Permission（权限）

### 为什么需要权限系统？

让 AI 执行 shell 命令是强大的，但也是危险的。一条 `rm -rf /` 就够你喝一壶的。Claude Code 的权限系统就是你的安全阀。

### 三种权限模式

**1. Interactive（交互模式，默认）**

每次写文件或执行命令前都会询问你：

```
Claude wants to execute: npm install express
Allow? [y/n/a]
  y - 允许这一次
  n - 拒绝
  a - 允许同类操作（本 Session 内）
```

**2. Auto-approve（自动批准）**

对已知安全的操作自动放行，对危险操作仍然询问。可以在配置中定义白名单：

```json
{
  "permissions": {
    "allow": [
      "read:**",
      "exec:npm test",
      "exec:npm run lint"
    ],
    "deny": [
      "exec:rm -rf*",
      "exec:sudo*"
    ]
  }
}
```

**3. Bypass（跳过模式）**

所有操作自动执行，不询问。**仅推荐在 CI/CD 或完全信任的场景下使用：**

```bash
claude --permission-mode bypassPermissions --print "运行所有测试"
```

### 权限最佳实践

1. **日常开发用 Interactive**：养成"看一眼再放行"的习惯
2. **团队脚本用 Auto-approve + 白名单**：只放行需要的命令
3. **CI/CD 用 Bypass + 沙箱**：在容器中运行，限制文件系统访问
4. **永远不要在生产环境的主机上用 Bypass**

## 四个概念如何协同工作

想象一个实际场景：你要给项目添加用户认证功能。

1. **Session 启动**：`claude` → 新 Session 创建
2. **Context 加载**：自动读取 CLAUDE.md，理解项目结构和技术栈
3. **你下达指令**："添加 JWT 用户认证，支持注册和登录"
4. **Tools 执行**：
   - `read` 读取现有的路由和模型文件
   - `exec` 安装 `pyjwt` 和 `passlib` 依赖
   - `write` 创建 auth 模块的代码文件
   - `edit` 修改主路由文件注册新端点
   - `exec` 运行测试验证功能正常
5. **Permission 控制**：每一步写入和执行都经过你的确认

这四个概念各司其职，共同构成了一个**安全、透明、高效**的 AI 编程体验。

## 小结

- **Session** = 一次完整的工作对话，有状态、可恢复
- **Context** = AI 当前的知识范围，通过 CLAUDE.md 主动管理
- **Tools** = AI 的行动能力，可扩展
- **Permission** = 安全保障，分级控制
