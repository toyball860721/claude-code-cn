# 第 4 章：Skills 系统 — 如何写自定义 Skills

## 什么是 Skills？

Skills 是 Claude Code 的扩展能力系统。简单来说，一个 Skill 就是一个 Markdown 文件（SKILL.md），里面用自然语言描述了一个特定任务的执行方式。当你的指令匹配到某个 Skill 的描述时，Claude Code 会自动加载并按照 Skill 中的指引来工作。

你可以把 Skill 理解为"给 AI 写的标准操作手册"——告诉它在特定场景下应该怎么做、用什么工具、遵循什么规范。

## Skill 的基本结构

一个标准的 SKILL.md 文件包含以下部分：

```markdown
# Skill 名称

> 一句话描述这个 Skill 做什么

## 触发条件
什么时候应该使用这个 Skill

## 输入
需要什么信息或参数

## 执行步骤
1. 第一步做什么
2. 第二步做什么
3. ...

## 输出格式
结果应该是什么样的

## 注意事项
边界条件、限制、最佳实践
```

## Skill 的存放位置

Skills 可以放在多个位置，Claude Code 会按优先级加载：

1. **项目级别**：`your-project/.claude/skills/` — 仅对当前项目生效
2. **用户级别**：`~/.claude/skills/` — 对你所有项目生效
3. **OpenClaw Skills**：通过 OpenClaw 安装的全局 Skills

## 实例 1：中文 Git Commit 生成器

```markdown
# 中文 Git Commit 生成器

> 分析代码变更，自动生成符合规范的中文 git commit message

## 触发条件
当用户说"提交代码"、"生成 commit"、"写 commit message"时触发

## 执行步骤
1. 运行 `git diff --staged` 查看暂存的改动
2. 如果没有暂存的改动，运行 `git diff` 查看工作区改动，并提示用户先 git add
3. 分析改动的类型和范围
4. 按以下格式生成 commit message：

## 输出格式
```
<类型>(<范围>): <简短描述>

<详细说明（可选）>
```

类型包括：
- feat: 新功能
- fix: 修复 bug
- docs: 文档更新
- style: 代码格式
- refactor: 重构
- test: 测试相关
- chore: 构建/工具

## 注意事项
- 简短描述不超过 50 个汉字
- 详细说明使用中文
- 如果改动涉及多个范围，使用逗号分隔
```

## 实例 2：API 文档生成器

```markdown
# API 文档生成器

> 扫描代码中的 API 端点，自动生成 OpenAPI/Swagger 格式的文档

## 触发条件
当用户说"生成 API 文档"、"更新接口文档"时触发

## 执行步骤
1. 扫描路由文件（Express: routes/, FastAPI: routers/, Spring: controller/）
2. 提取每个端点的：HTTP 方法、路径、参数、请求体、响应体
3. 读取相关的类型定义/数据模型
4. 生成 Markdown 格式的 API 文档

## 输出格式
对每个端点生成：
- 端点名称和描述
- 请求方法和路径
- 请求参数表格（名称、类型、是否必填、说明）
- 请求体 JSON 示例
- 响应体 JSON 示例
- 错误码说明

## 注意事项
- 自动检测框架类型
- 如果代码中有 JSDoc/docstring，优先使用其中的描述
- 生成的文档保存到 docs/api.md
```

## 实例 3：代码审查助手

```markdown
# 代码审查助手

> 对指定文件或 PR 进行全面的代码审查，输出中文审查报告

## 触发条件
当用户说"审查代码"、"code review"、"CR"时触发

## 执行步骤
1. 确定审查范围：
   - 如果指定了文件路径，审查该文件
   - 如果说"审查 PR"，运行 `git diff main...HEAD` 获取改动
   - 如果没有指定，审查最近一次 commit 的改动
2. 对每个改动文件进行分析
3. 从以下维度评审：
   - 🐛 Bug 风险：可能的运行时错误、边界情况
   - 🔒 安全性：注入、越权、信息泄露
   - ⚡ 性能：N+1 查询、内存泄漏、不必要的计算
   - 📖 可读性：命名、注释、代码组织
   - 🧪 测试覆盖：是否有对应的测试

## 输出格式
### 审查报告
**总体评价**：✅ 可合并 / ⚠️ 建议修改 / ❌ 需要重写

#### 问题列表
| # | 文件 | 行号 | 级别 | 问题描述 | 建议修复 |
|---|------|------|------|----------|----------|
| 1 | ... | ... | 🔴严重 | ... | ... |

#### 亮点
（好的代码实践也要指出来）
```

## 实例 4：单元测试生成器

```markdown
# 单元测试生成器

> 为指定的函数或模块自动生成全面的单元测试

## 触发条件
当用户说"写测试"、"生成测试"、"补充测试"时触发

## 执行步骤
1. 读取目标文件，分析导出的函数/类
2. 检测项目使用的测试框架（Jest/Vitest/Pytest/Go test 等）
3. 为每个函数生成测试用例，覆盖：
   - 正常路径（happy path）
   - 边界条件（空值、零值、最大值）
   - 错误路径（异常输入、网络错误）
   - 类型边界（TypeScript 的类型守卫场景）
4. 运行测试，修复失败的用例
5. 检查覆盖率

## 输出
- 测试文件保存到对应的 __tests__/ 或 tests/ 目录
- 打印覆盖率报告
```

## 实例 5：数据库迁移助手

```markdown
# 数据库迁移助手

> 根据模型变更自动生成数据库迁移文件

## 触发条件
当用户说"生成迁移"、"数据库迁移"、"migration"时触发

## 执行步骤
1. 检测 ORM 类型（Prisma/SQLAlchemy/TypeORM/Drizzle 等）
2. 比较当前模型定义与上一次迁移的差异
3. 生成迁移文件
4. 如果是 Prisma：运行 `npx prisma migrate dev --name <描述>`
5. 如果是 Alembic：运行 `alembic revision --autogenerate -m "<描述>"`
6. 检查生成的迁移文件，确保：
   - 没有数据丢失风险
   - 大表的 ALTER TABLE 有注意性能影响
   - 有对应的回滚操作

## 注意事项
- 生产环境的迁移要格外小心，提醒用户备份
- 添加索引的迁移建议使用 CONCURRENTLY（PostgreSQL）
- 列类型变更需要确认数据兼容性
```

## 编写 Skill 的最佳实践

### 1. 描述要具体

**好的描述：**"扫描 Python 项目中的所有 FastAPI 路由，提取端点定义，生成 Markdown 格式的 API 文档"

**差的描述：**"生成文档"

### 2. 步骤要有容错

```markdown
## 执行步骤
1. 检查项目类型
   - 如果是 Node.js 项目（存在 package.json）→ 使用 npm/yarn
   - 如果是 Python 项目（存在 pyproject.toml 或 requirements.txt）→ 使用 pip
   - 如果无法判断 → 询问用户
```

### 3. 输出格式要明确

不要说"输出结果"，要说"以 Markdown 表格格式输出，包含以下列：..."。

### 4. 加入验证步骤

好的 Skill 应该包含自我验证：生成代码后运行测试，生成配置后验证语法，写了文件后检查格式。

### 5. 定义边界

明确说明 Skill 不做什么，避免越界操作：

```markdown
## 不做的事
- 不直接修改数据库（只生成迁移文件）
- 不删除现有测试（只添加新测试）
- 不修改 CI/CD 配置（提供建议但需人工确认）
```

## 小结

Skills 系统是 Claude Code 的核心竞争力之一。通过编写 Skills，你可以：

- 将团队的最佳实践编码为可重复的流程
- 降低新成员的上手门槛
- 确保 AI 的输出符合你的标准和规范
- 不断积累和迭代你的"AI 工具箱"
